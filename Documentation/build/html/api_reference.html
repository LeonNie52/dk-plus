API Reference¶

class drone_network.Networking(address, protocol, vehicle)¶

Tip

In this class you can add your own networking protocol method. See Networking.create_udp_broadcast for an example.

The necessary class to interface your drone with the outer world.

  • Responsible for opening/closing sockets according to a chosen protocol (currently only broadcast messaging through UDP).
  • Invokes the send/receive/task threads.
  • Contains the vehicle’s drone list and message queue.
  • Invokes the necessary parameter observers/listeners
Parameters:address (tuple) – The network address in (‘address’, ‘port’) format.
Parameters:
  • protocol (str) – The desired network protocol as implemeted inside the class.
  • vehicle (dronekit.Vehicle) – The object vehicle as returned from dronekit.connect().
POLL_RATE¶

Set to 0.5 seconds. How often to send the messages (in seconds).

MAX_STAY¶

Set to 5 seconds. How long should messages be kept for without update (in seconds).

SAFETY_ZONE¶

Set to 40 metres. The radius in metres around the drone that is considered the border of the safety zone.

CRITICAL_ZONE¶

Set to 10 metres. The radius in metres around the drone that is considered the border of the critical zone.

msg_queue¶

A Queue.Queue object for storing received messages. Can be changed directly.

vehicle_params¶

A params.Params object to save the wanted parameters. It is not advised to modify it directly since the structure is updated automatically.

drones¶

A list that saves the drone parameters of incoming drone messages. It is not advised to modify it directly since the structure is updated automatically.

context¶

A dictionary that helps save important information just before any collision avoidance action.

Note

This attribute is not used in current version. You can use context instead.

priority¶

An int that indicates the drone’s priority number in case of a collision avoidance scheme. It is currently not used but may prove helpful for future versions.

sock_send¶

A socket.socket object used for sending messages.

sock_receive¶

A socket.socket object used for receiving messages.

t_send¶

A SendThread object that initializes the threading responsible for sending the drone parameters.

t_receive¶

A ReceiveThread object that initializes the threading responsible for receiving the other drones’ parameters and adding them to msg_queue.

t_task¶

A ReceiveTaskThread object that initializes the threading responsible for processing the drone parameters received.

run()¶
  • Opens the network sockets depending the explicitly stated protocol
  • Starts the threads responsible for sending (t_send) and receiving messages (t_receive, t_task).
stop()¶

Closes the socket objects, opened by Networking.run().

create_udp_broadcast(address)¶

Opens two UDP sockets, one for sending and one for receiving messages. The sockets are set for broadcast messages by invoking the socket.SO_BROADCAST flag.

Parameters:address (tuple) – The network address in (‘address’, ‘port’) format.
Returns:exit_status = ‘1’ if the method performs succesfully all the tasks, ‘0’ otherwise.
Return type:int
Raises:socket.error – When failing to run any socket commands
populate_drones(scenario)¶

A method that populates the drones list for testing purposes.

Parameters:scenario (str) – A string specifying the wanted scenario. Must be implemented inside the method.
class send_thread.SendThread(threading.Thread, network)¶

A subclass of threading.Thread. A thread that when it is started, is responsible for sending the drone’s vehicle parameters at the address specified when creating a Networking instance. It is self invoked inside the Networking class.

The class needs a socket to be open for sending messages.

Params:network: A Networking object.
daemon = True

It is set to True so that the thread can keep running until the main thread finishes

run()¶

An infinite while loop that pickles the vehicle_params attribute and sends it as a message to the specified socket. time.sleep is invoked with the POLL_RATE parameter in order to set the frequency of those messages.

Warning

Be careful when setting the frequency too high as this may flood the network and reduce performance. High frequency means small POLL_RATE number.

Raises:exception – When failing to send the message.
class receive_thread.ReceiveThread(threading.Thread, network)¶

A subclass of threading.Thread. A thread that when it is started, is responsible for receiving any incoming messages from the address specified when creating a Networking instance. It is self invoked inside the Networking class.

The class needs a bound socket to be open for receiving incoming messages.

Params:network: A Networking object.
daemon = True

It is set to True so that the thread can keep running until the main thread finishes

run()¶

An infinite while loop that awaits for incoming messages with the select() function. If the received message is coming from a drone, then it adds it to the msg_queue queue for further processing.

Raises:
  • pickle.UnpicklingError – if the receiving message is not pickled.
  • exception – if there is a problem with the sockets.
class receive_task_thread.ReceiveTaskThread¶

A subclass of threading.Thread. A thread that when it is started, is responsible for processing any messages in msg_queue queue. It is self invoked inside the Networking class.

Params:network: A Networking object.
daemon = True

It is set to True so that the thread can keep running until the main thread finishes

run()¶

An infinite while loop that processes the received messages one by one. Processing goes as follows:

  • Unpickle the message
  • Add a timestamp
  • Calculate the distance between the two drones by calling geo_tools.get_distance_metres
  • Reject message if distance is beyond the SAFETY_ZONE limit.
  • Add to drones list for further processing by the Collision Avoidance thread.

Note

Queue.Queue is thread-safe, so after finishing the task on the message, Queue.Queue.task_done has to be called.

Raises:Queue.Empty – If no messages are inside the msg_queue.
class collision_avoidance.CollisionThread(threading.Thread)¶

Handler for the collision avoidance part of the module. A specific Networking object needs to be running. Here is the whole API for collision avoidance matters like context switching, prioritization and control.

Tip

This is the class where you can deploy your own cooperative collision avoidance algorithm and test it. Check out this Guide.

near¶

A list that stores all the drones in the drones list that belong inside the SAFETY_ZONE but out of the CRITICAL_ZONE.

critical¶

A list that stores all the drones in the drones list that belong inside the CRITICAL_ZONE only.

in_session¶

It is True when a collision avoidance scheme is in action and False otherwise.

context¶

A collections.namedtuple to save pre-collision avoidance information. Currently what is saved is mode, mission and next waypoint.

run()¶

Based on the protocol that is used during the instantiation of a Networking object, it calls repeatedly the according method. It is like a switch-case piece of code for routing to the desired protocol.

Note

Currently the only supported protocol is the ‘naive priorities’ one. If you want to experiment with other protocols, they must be implemented as a method inside this class.

Warning

The time.sleep function must be called so that the main thread can acquire the lock, too.

no_protocol()¶

If no protocol is used then there is a simple near and critical lists update. Then the output of the nearby drones is printed on stdout by calling print_drones_in_vicinity().

priorities_protocol()¶

Based on this protocol, whenever there are nearby drones, each drone acquires explicitly a priority number. Every drone waits until its priority number becomes ‘1’, so they can start taking actions. Waiting means keeping the same position in all three axes (lat, lon, altitude).

Warning

Because of the high latency observed, a drone demonstrates a “momentum”. This is troublesome because the drone does not freeze instantly. Instead, it flies some metres away (depending the groundpseed) before it gradually stops. For this reason you have to be EXTRA CAREFUL when setting the SAFETY_ZONE attribute.

The protocol goes as follows:

  • give_priorities
  • Update near and critical lists
  • give_control or take_control depending the priority number

Note

CRITICAL_ZONE is not utilised so far but maybe it can prove useful in the future.

give_priorities()¶

The algorithm to spread the priority numbers to drones. Since each drone must hold the same algorithm, the prioritization is global.

Note

Some might naturally think that this algorithm has flaws. It has been tested under worst-case scenarios and has given good results. A possible worst-case scenario is when there are three drones in line but not all drones see all. You can easily see that this “chained-prioritization” works just fine with the ‘priorities’ protocol.

Priorities are given based on the following rules:

  • If drone is grounded then its priority is low, exempting mode ‘ARMED’
  • Then, if system status needs to be checked first, if it’s critical it has the highest priority of all
  • Then, if status is OK then highest priority is given to highest mission importance first
  • Then, among same mission importances, highest priority is given to the ones with less capabilities
  • Then, if they have the same status, mission importance, capabilities fine-tune with battery level
Returns:Updates the drones list with the correct priorities.
take_control()¶

Takes control from flight controller by changing the mode to POSHOLD and overriding the RC3 channel (throttle). If control is being taken for first time, then the state of the drone is first saved to context.

Warning

  • FIXME: The “momentum” until the drone finally stops lasts for about 3-4 seconds. During this time, if initial speed is 4-6 m/s, the drone can stop more than 10 metres off the initial take_control call.
  • FIXME: If previous mode was POSHOLD then context saving will not happen. POSHOLD is a manual mode though, so most probably there will be no commands to be executed.

Note

Defensive coding has to be performed for the change of mode since commands by the flight controller are dropped silently.

give_control()¶

Gives control to flight controller by restoring the drone’s previous state and cancelling the RC3 channel override. The in_session variable is returned to False to signal the end of the collision avoidance actions.

Use case: If the mode before was ‘AUTO’ with loaded waypoints, then the drone will continue its mission from where it stopped.

save_context()¶

Invoked at the first time of take_control() call. All information is saved in the context class attribute.

What is currently saved is:

  • Mode
  • Mission (loaded waypoints)
  • Next waypoint

Tip

You are free to change the information changed according to your own collision avoidance protocol. You can see all the parameters that can be saved at the DK-Python API reference and the MAVLink messages (from line 2095).

restore_context()¶

Restores drone’s saved state.

Tip

In order to load the saved commands, the vehicle has to be in GUIDED mode.

The procedure goes as follows:

  • Change to GUIDED mode
  • Clear all the uploaded commands
  • Upload the previous commands
  • Change mode and next_waypoint

Note

Saving/Restoring the context does not have any meaning in the ‘priorities’ protocol, since no extra actions are given to the controller, just an RC override. These functions are useful if you want to set a different flight plan for your collision avoidance protocol. They are invoked inside the ‘priorities’ protocol for you to see that they are functional.

update_drone_list()¶

This function is responsible for:

  • Cleaning-up drones that have not given a heartbeat the last MAX_STAY seconds.
  • Updating the lists (quite pythonically) containing the drones that are inside the SAFETY_ZONE and the CRITICAL_ZONE
print_drones_in_vicinity()¶

A simple function that prints to stdout the drones in the near and critical lists.

current_mission()¶

A simple function that retrieves the commands uploaded to the controller. See more information here.

Returns:The mission object. See restore_context() for an example of how to access its variables.
Return type:CommandSequence
get_priority_num()¶

Accesses the drones list (should be called after the give_priorities() method) in order to get the vehicle’s priority number.

Returns:The priority number of the vehicle.
Return type:int
class params.Params([network=None, vehicle=None, dummy=False])¶

This class provides all the necessary vehicle attributes for the collision avoidance handling. All these parameters are updated automatically according to a threshold that is defined separately for every parameter. The parameters are sent every POLL_RATE seconds if a SendThread object is running.

Parameters:
  • network (Networking) – The Networking object.
  • vehicle (Vehicle) – The dronekit.Vehicle object.
  • dummy (bool) – If set to True then dummy parameters are added, for testing purposes.
ID¶

A random ID, unique for every drone running the code. Uniqueness is guaranteed by running the uuid.uuid4 function. Please do not change this value by yourself.

last_recv¶

A timestamp that is given upon receiving the specific drone’s parameters by ReceiveTaskThread.

version¶

The release version of dronekit that runs the vehicle.

ekf_ok¶

bool value that indicates EKF functionality.

gps_fix¶

An int value that indicates the vehicle’s GPS fix. Must be ‘3’ in order to work properly.

gps_sat¶

int value that specifies the number of satellites the GPS receives from (ideally 4 or more).

gps_eph¶

The EPH (horizontal accuracy).

gps_epv¶

The EPV (vertical accuracy).

set_global_alt¶

bool value that indicates if altitude in the global frame can be changed offboard.

set_attitude¶

bool value that indicates if attitude (pitch, yaw, roll) can be changed offboard.

mode¶

Current mode of vehicle. Access mode.name for an str representation.

global_alt¶

Current altitude in the global frame.

global_lat¶

Current latitude in the global frame.

global_lon¶

Current longititude in the global frame.

distance_from_self¶

How many metres away is the received drone from this vehicle. Set upon receiving the specific drone’s parameters by ReceiveTaskThread.

mission_importance¶

Default value is 0. It is changed depending the use (check Mission Importance Setter for more information)

  • Level 0: for everyday users and hobbyists
  • Level 1: for commercial users/companies
  • Level 2: governmental missions, search&rescue, national security
heading¶

Where the drone is heading, in degrees, with 0 degrees being the North.

next_wp¶

Next waypoint number as obtained from uploaded commands.

next_wp_lat¶

Next waypoint’s global latitude.

next_wp_lon¶

Next waypoint’s global longitude.

next_wp_alt¶

Next waypoint’s global altitude.

battery_level¶

Battery level in percentage (100% is full).

velocity¶

3-axis velocity in m/s.

groundspeed¶

Groundspeed in m/s.

airspeed¶

Airspeed in m/s. It can be different from groundspeed because of location’s air speed etc.

system_status¶

Must be ‘OK’ to be able to fly. There are other statuses that indicate that something is wrong like ‘EMERGENCY’, ‘CRITICAL’ etc.

Tip

Some attributes might not be readily available just by accesing the dronekit.Vehicle object. One example is the signal that the next waypoint has changed. If DK-Python does not mention the attribute you want to observe/send, you will have to check the MAVLink message list and use the @vehicle.on_message decorator.

Example: Checking if the next waypoint has changed. If it has changed print the destination coordinates and assign them to the vehicle’s parameters for easier access`:

@vehicle.on_message('MISSION_CURRENT')
def message_listener(self, name, message):
        try:
                if network.vehicle_params.next_wp == message.seq:
                        return
                else:
                        print 'Next waypoint changed'
                        network.vehicle_params.next_wp = message.seq

                        cmd = vehicle.commands

                        if cmd.count == 0:
                                print 'No waypoints found'

                        else:
                                print 'Waypoint', cmd.next, ' out of ', cmd.count, ':'
                                pos = cmd.next - 1

                                print 'Frame, Lat, Lon, Alt:', cmd[pos].frame, cmd[
                                pos].x, cmd[pos].y, cmd[pos].z
                                network.vehicle_params.next_wp_lat = cmd[pos].x
                                network.vehicle_params.next_wp_lon = cmd[pos].y
                                network.vehicle_params.next_wp_alt = cmd[pos].z
        except Exception, e:
                print "Error: ", e
add_listeners(network, vehicle)¶

Uses function decorators to observe the attributes in the Params class. See the source code or read this Guide for further information. Note that changes are thresholded in order to minimize writes. Thresholding has been done based on own experience, observations and for certain purposes.

Warning

If you add a new attribute in this class then you will have to add its observer in this function, too. Otherwise the value will not be updated and will send wrong information about it.

print_all()¶

Prints to stdout all the attributes observed in the Params class.

mission_importance(level)

A setter function. This attribute must not be able to be changed by the user for safety reasons. Mission importance must have the following semantics:

  • Level 0: for everyday users and hobbyists
  • Level 1: for commercial users/companies
  • Level 2: governmental missions, search&rescue, national security

Warning

Since the language is Python and the script can be changed by anyone, it is not safe to assume that the mission_importance attribute performs its role. Server-based authentication has to be deployed in order to verify the change in mission_importance. Do NOT utilise it as it is now.

class geo_tools¶

A set of functions with code from here and there for geographical transformations.

get_location_metres(lat, lon, alt, dNorth, dEast)¶

Returns the latitude/longitude dNorth and dEast metres from the specified original location. The returned location has the same alt value as the original.

The function is useful when you want to move the vehicle around specifying locations relative to the current vehicle position. The algorithm is relatively accurate over small distances (10m within 1km) except close to the poles. For more information see: http://gis.stackexchange.com/questions/2951/algorithm-for-offsetting-a-latitude-longitude-by-some-amount-of-meters

earth_radius=6378137.0

According to theory.

Parameters:
  • lat (float) – Current global latitude in decimal degrees
  • lon (float) – Current global longitude in decimal degrees.
  • alt (float) – Current relative altitude in decimal degrees.
  • dNorth (int) – North offset in metres.
  • dEast (int) – East offset in metres.
Returns:

LocationGlobal: The new offset coordinates in decimal degrees.
Return type:

tuple(lat, lon, alt)
get_distance_metres(lat1, lon1, lat2, lon2)¶

Returns the ground distance in metres between two coordinate sets. This method is an approximation, and will not be accurate over large distances and close to the earth’s poles. It comes from the ArduPilot test code: https://github.com/diydrones/ardupilot/blob/master/Tools/autotest/common.py

Parameters:
  • lat1 (float) – First latitude in global frame in decimal degrees.
  • lon1 (float) – First longitude in global frame in decimal degrees.
  • lat2 (float) – Second latitude in global frame in decimal degrees.
  • lon2 (float) – Second longitude in global frame in decimal degrees.
Returns:

Ground distance between coordinates in metres
Return type:

float

Returns:Distance to current waypoint in metres.
